## 目的

为云端查询服务提供自建的轻量级防刷机制，防止简单脚本（curl/requests）批量调用接口、爬虫遍历呼号抓取数据以及自动化批量绑定虚假呼号。由于 QSL 查询服务攻击价值较低，不需要应对专业的验证码破解服务，采用自建轻量方案即可。
## 需求
### 需求：IP 限流

系统必须对所有 API 请求按 IP 进行频率限制，默认限制为 20 次/分钟。超限请求必须返回 HTTP 429 状态码及友好提示。计数必须使用 Cloudflare KV 存储，支持自动过期。计数键中的 IP **必须**取自「可信真实客户端 IP 解析」（见 `trusted-client-ip` 规范）得到的真实用户 IP，**禁止**直接使用 `CF-Connecting-IP` 或任何客户端可注入头作计数键——在前置阿里云 CDN 架构下，经 CDN 回源时 `CF-Connecting-IP` 是 CDN 回源节点 IP（把大量真实用户归并到少数桶、限流粒度失真），须改取经密钥回源头（`X-Origin-Auth`）校验、采信 CDN 写入的不可伪造真实 IP 头的可信解析，方能在直连与经 CDN 两种入口下都按真实用户计数。

> 说明：本「IP 限流」是抬高自动化批量调用成本（anti-abuse），非访问控制；其有效性依赖计数键能区分真实用户。CDN 路径下若按 CDN 节点 IP 计数则形同虚设（见 `trusted-client-ip`）。

#### 场景：请求频率超过限制

- **当** 同一真实用户 IP 在一分钟内的 API 请求次数超过 20 次
- **那么** 系统拒绝该请求并返回 HTTP 429 状态码及友好提示

#### 场景：请求频率在限制内

- **当** 同一真实用户 IP 在一分钟内的 API 请求次数不超过 20 次
- **那么** 系统使用 Cloudflare KV 存储记录计数（支持自动过期）并放行该请求

#### 场景：CDN 路径下按真实用户 IP 计数

- **当** 请求经阿里云 CDN 回源到达受限流的 API 端点
- **那么** 计数键中的 IP **必须**为经密钥回源头校验后采信的真实用户 IP（取自 CDN 写入的不可伪造头）
- **并且** **禁止**将不同真实用户归并到同一 CDN 回源节点 IP 的计数桶
- **并且** 客户端伪造 `X-Forwarded-For` 经 CDN 透传后**禁止**绕过该限流

### 需求：请求签名校验

所有 API 请求必须携带由前端 JS 生成的有效签名，签名包含时间戳、随机数（nonce）和 SHA-256 哈希。服务端必须校验签名有效性、时间窗口（5 分钟内）和 nonce 唯一性；nonce 必须使用 KV 存储去重，并设置 5 分钟 TTL 自动过期。签名无效或过期的请求必须返回 HTTP 403。

#### 场景：携带有效签名的请求

- **当** API 请求携带有效签名，且时间戳在 5 分钟窗口内，nonce 未被使用过
- **那么** 服务端校验通过并放行请求，同时将该 nonce 写入 KV（设置 5 分钟 TTL 自动过期）

#### 场景：签名无效或过期的请求

- **当** API 请求的签名无效、时间戳超出 5 分钟窗口，或 nonce 已被使用过
- **那么** 服务端拒绝该请求并返回 HTTP 403

### 需求：算术验证码

「订阅收卡」操作触发前必须完成算术验证码。验证码为简单加法或减法运算（如 "12 + 7 = ?"），题目必须使用 Canvas 渲染并包含轻微干扰线以增加 OCR 识别难度。服务端生成题目时必须返回加密 token，包含正确答案和过期时间；用户提交答案时，服务端必须校验 token 有效性和答案正确性。token 有效期为 5 分钟，过期需重新获取题目。

#### 场景：触发订阅收卡操作

- **当** 用户触发「订阅收卡」操作
- **那么** 系统要求先完成算术验证码；服务端生成简单加减法题目（如 "12 + 7 = ?"），以 Canvas 渲染并附带轻微干扰线，并返回包含正确答案与过期时间的加密 token

#### 场景：提交正确且未过期的答案

- **当** 用户在 token 5 分钟有效期内提交答案，且 token 有效、答案正确
- **那么** 服务端校验通过，允许订阅收卡操作继续

#### 场景：token 过期或答案错误

- **当** 用户提交答案时 token 已超过 5 分钟有效期，或答案不正确
- **那么** 服务端校验失败，要求用户重新获取题目并作答

### 需求：用户体验

签名校验必须对用户完全无感。算术验证码仅在订阅操作时出现，查询操作无需交互。验证码 Canvas 必须在移动端清晰可读。

#### 场景：用户执行查询与订阅操作

- **当** 用户执行查询操作或订阅操作
- **那么** 查询操作无需任何验证交互且签名校验对用户完全无感，仅订阅操作出现算术验证码，且验证码 Canvas 在移动端清晰可读

### 需求：性能

限流检查延迟必须 < 10ms（KV 读取），签名校验延迟必须 < 5ms（纯计算），验证码生成延迟必须 < 20ms。

#### 场景：处理防护相关请求

- **当** 系统执行限流检查、签名校验或验证码生成
- **那么** 限流检查延迟 < 10ms（KV 读取），签名校验延迟 < 5ms（纯计算），验证码生成延迟 < 20ms

### 需求：安全

签名密钥必须存储在环境变量中，前端仅获取用于签名的公开部分。验证码 token 必须使用服务端密钥签名以防止伪造。系统禁止记录用户 IP 的完整日志，IP 仅用于限流计数。

#### 场景：处理密钥、token 与用户 IP

- **当** 系统签发签名密钥、生成验证码 token 或处理用户 IP
- **那么** 签名密钥仅存于环境变量（前端只获取公开部分），验证码 token 使用服务端密钥签名以防伪造，且不记录用户 IP 的完整日志（仅用于限流计数）

### 需求：请求签名格式

所有需要签名的请求必须携带以下参数：

| 参数 | 类型 | 说明 |
|------|------|------|
| `_ts` | number | 请求时间戳（毫秒） |
| `_nonce` | string | 随机字符串（UUID） |
| `_sig` | string | SHA-256 签名 |

签名算法必须为：

```
payload = `${请求路径}:${关键参数}:${_ts}:${_nonce}`
signature = sha256(payload + CLIENT_SIGN_KEY)
```

#### 场景：构造需签名的请求

- **当** 前端构造一个需要签名的请求
- **那么** 请求必须携带 `_ts`（毫秒时间戳）、`_nonce`（UUID 随机字符串）与 `_sig`（SHA-256 签名）
- **并且** `_sig` 由 `payload = ${请求路径}:${关键参数}:${_ts}:${_nonce}` 经 `sha256(payload + CLIENT_SIGN_KEY)` 计算得出

### 需求：验证码生成接口

系统必须提供 `GET /api/captcha` 接口用于生成算术验证码题目，响应包含 `question`、`token` 与 `expires` 字段。`question` 字段仅供调试或无障碍场景使用，正常情况下前端应使用 Canvas 渲染。

#### 场景：请求生成验证码题目

- **当** 客户端请求 `GET /api/captcha`
- **那么** 服务端返回包含 `question`（如 "12 + 7 = ?"）、`token`（如 "eyJ..."）和 `expires`（过期时间戳）的 JSON 响应
- **并且** `question` 字段仅供调试或无障碍场景使用，正常情况下前端应使用 Canvas 渲染

### 需求：验证码校验参数

订阅接口必须额外携带验证码校验参数：

| 参数 | 类型 | 说明 |
|------|------|------|
| `captcha_token` | string | 验证码 token |
| `captcha_answer` | number | 用户输入的答案 |

#### 场景：提交订阅请求

- **当** 客户端提交订阅接口请求
- **那么** 请求必须额外携带 `captcha_token`（string，验证码 token）与 `captcha_answer`（number，用户输入的答案）

### 需求：范围限制

本能力禁止包含以下内容：图形验证码（点选、滑动拼图）、第三方验证码服务集成（腾讯天御、极验等）、行为分析 / 风控系统、IP 黑名单持久化管理。

#### 场景：评估超出范围的防护手段

- **当** 评估是否引入图形验证码、第三方验证码服务、行为分析/风控系统或 IP 黑名单持久化管理
- **那么** 这些手段均不在本能力范围内，不予实现
