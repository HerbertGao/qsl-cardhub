## 需求
### 需求：数据导出功能

系统**必须**支持将本地数据库导出为 JSON 文件，便于备份和迁移。

#### 场景：导出全部数据

**前提条件：**
- 本地数据库已初始化

**操作步骤：**
1. 用户打开"数据管理"页面
2. 用户点击"导出数据"按钮
3. 系统弹出文件保存对话框
4. 用户选择保存位置和文件名
5. 用户确认保存

**预期结果：**
- 文件扩展名为 `.qslhub`
- 文件格式为 JSON（UTF-8 编码）
- 文件包含元信息：
  - `version`: 导出格式版本（"1.1"）
  - `db_version`: 数据库版本号（整数）
  - `db_version_display`: 可读版本号（如 "2026.1.23.003"）
  - `app_version`: 应用版本号
  - `exported_at`: 导出时间戳（ISO 8601 格式）
  - `client_id`: 云端同步客户端标识（可选，仅当已配置同步时包含）
- 文件包含所有表的数据：
  - `tables.projects`: 项目列表
  - `tables.cards`: 卡片列表（必须包含 serial 字段）
  - `tables.sf_senders`: 顺丰寄件人列表
  - `tables.sf_orders`: 顺丰订单列表
  - `tables.app_settings`: 全局配置项列表（可选字段，向后兼容）
- 导出成功后显示统计信息（项目数、卡片数、寄件人数、订单数）
- 默认文件名格式：`qslhub_backup_YYYYMMDD_HHmmss.qslhub`

#### 场景：导出包含 client_id

- **当** 用户已配置云端同步（sync.toml 存在且包含 client_id）
- **那么** 导出文件的顶层必须包含 `client_id` 字段，值为当前同步配置的 client_id

#### 场景：未配置同步时导出

- **当** 用户未配置云端同步（sync.toml 不存在）
- **那么** 导出文件的顶层 `client_id` 字段必须为 null 或不存在

#### 场景：空数据库导出

**前提条件：**
- 本地数据库已初始化但无数据

**操作步骤：**
1. 用户点击"导出数据"

**预期结果：**
- 导出成功
- 文件中表数据为空数组
- 版本信息正确

### 需求：数据导入功能

系统**必须**支持从 JSON 文件导入数据到本地数据库，并进行版本兼容性检查。导入时**必须**完整恢复所有字段，包括卡片的 serial（序号）字段。

#### 场景：导入兼容版本数据

**前提条件：**
- 导入文件的 `db_version` <= 本地数据库版本
- 导入文件格式正确

**操作步骤：**
1. 用户打开"数据管理"页面
2. 用户点击"导入数据"按钮
3. 系统弹出文件选择对话框（过滤 `.qslhub` 文件）
4. 用户选择文件
5. 系统显示导入预览对话框：
   - 文件版本信息
   - 导出时间
   - 数据统计（项目数、卡片数等）
   - 版本兼容性状态
6. 系统显示覆盖警告："导入将覆盖本地所有数据，此操作不可逆"
7. 用户确认导入

**预期结果：**
- 本地数据被清空（事务内）
- 导入文件中的数据写入数据库（事务内）
- 卡片的 serial 字段必须正确写入数据库，禁止遗漏
- 如果导入文件包含 `tables.app_settings`，必须清空并恢复 `app_settings` 表数据
- 如果导入文件不包含 `tables.app_settings`（旧版文件），必须保留现有 `app_settings` 数据不变
- 如果发生错误，事务回滚，本地数据不变
- 导入成功后显示统计信息
- 页面数据自动刷新

#### 场景：导入包含 client_id 的文件

- **当** 导入文件的顶层包含非空 `client_id` 字段
- **那么** 系统必须将该 `client_id` 写入本地 sync.toml 配置
- **并且** 必须保留 sync.toml 中已有的 `api_url` 和 `last_sync_at` 不变
- **并且** 如果 sync.toml 不存在，必须创建包含该 `client_id` 的默认配置

#### 场景：导入不含 client_id 的旧版文件

- **当** 导入文件不包含 `client_id` 字段或其值为 null
- **那么** 系统必须跳过 client_id 恢复，不修改本地 sync.toml
- **并且** 导入流程正常完成，不报错

#### 场景：拒绝高版本数据导入

**前提条件：**
- 导入文件的 `db_version` > 本地数据库版本

**操作步骤：**
1. 用户选择高版本的 `.qslhub` 文件

**预期结果：**
- 预览对话框显示错误状态
- 错误信息："导入文件的数据库版本（X.X.X.XXX）高于本地版本（Y.Y.Y.YYY），请升级应用后再导入"
- "确认导入"按钮禁用
- 本地数据不受影响

#### 场景：导入格式错误的文件

**前提条件：**
- 选择的文件不是有效的 JSON 或缺少必要字段

**操作步骤：**
1. 用户选择格式错误的文件

**预期结果：**
- 显示错误提示："文件格式错误，请选择有效的 QSL-CardHub 导出文件"
- 本地数据不受影响

#### 场景：导入过程中发生错误

**前提条件：**
- 导入文件格式正确
- 数据插入过程中发生错误（如外键约束失败）

**操作步骤：**
1. 用户确认导入
2. 系统开始导入
3. 导入过程中发生错误

**预期结果：**
- 事务回滚
- 本地数据保持不变
- 显示具体错误信息

### 需求：云端数据同步

系统**必须**支持将本地数据全量同步到用户自建的云端 API，并参与服务端的乐观并发版本护栏（见 `cloud-sync-versioning`）：持久化并回传 `base_version`、成功后刷新基线、遇 409 引导用户下载或强制覆盖。

系统**必须**支持用户申报所属租户（tenant 代码），并在云端请求中以请求头 `X-Tenant-Id` 携带该申报值，供服务端交叉校验（见 `tenant-isolation` / `cloud-backend-api`）。租户归属的真源**永远**是写凭据（`key→tenant`），`X-Tenant-Id` 仅用于申报与校验，**禁止**被客户端或服务端当作写入/读取的归属目标。租户代码为**软约束**：未填写时客户端**禁止**发送 `X-Tenant-Id`，行为与未引入本能力前逐字一致（服务端缺头向后兼容放行）。

#### 场景：配置云端 API

**前提条件：**
- 用户已部署云端接收服务（或选用官方云）
- 用户已获取 API 地址和 API Key

**操作步骤：**
1. 用户打开"数据管理"页面
2. 用户在"租户 & 云端同步"区域的「API 地址」下拉框选择**官方云预设**或**手动输入**自定义 API 地址
3. 用户输入 API Key
4. 用户输入租户代码（可选；输入完毕失焦时自动格式化）
5. 用户点击"保存配置"

**预期结果：**
- 「API 地址」**必须**是可手动输入的下拉框，预设至少两项：空值（未配置）与官方云地址；选择预设或手填均可
- 选中**官方云预设**时，租户代码与 API Key **必须**为必填（API Key 已保存时可留空保持不变）；其它地址（自托管）保持软约束、不强制
- API 地址保存到配置文件
- API Key 加密存储（使用凭据加密存储系统）；存储键名：`qsl-cardhub:sync:api_key`
- 租户代码保存到配置文件（`sync.toml` 的 `tenant` 字段）；留空时该字段为空/缺省，旧 `sync.toml`（无 `tenant` 字段）解析后**必须**为空（向后兼容）
- 显示保存成功提示
- 首次保存时自动生成 client_id（UUID 格式）

#### 场景：租户代码输入完毕自动格式化

**前提条件：**
- 用户在配置表单的租户代码输入框输入内容

**操作步骤：**
1. 用户输入含大写字母/空格/非法字符/超长的内容（如 `BH2RO`、`My Tenant`）
2. 输入框失焦（输入完毕）

**预期结果：**
- 系统**必须**在失焦时自动把租户代码格式化为合法 slug：去首尾空白 → 转小写 → 去除非 `[a-z0-9-]` 字符 → 截断至 32 位（如 `BH2RO`→`bh2ro`、`My Tenant`→`mytenant`）
- 格式化为**可见**变更（当面发生于输入框），不是保存时的静默转换
- 命令层保存时**仍必须**以 `^[a-z0-9-]{1,32}$` 做权威校验作兜底（格式化后应恒通过）

#### 场景：导出/导入同步配置（设备间迁移）

**前提条件：**
- 用户已配置云端 API（含 API Key）

**操作步骤：**
1. 用户点击"一键复制"
2. 在另一处/设备点击"一键粘贴"

**预期结果：**
- 「一键复制」**必须**把当前 API 地址、租户代码、API Key 导出为 Base64 编码字符串并写入剪贴板；导出串**含明文 API Key**（Base64 仅编码、非加密），界面**必须**提示其敏感性、仅供本机设备间迁移
- 导出**禁止**包含 `client_id`（设备本地自动生成、不随配置迁移）
- 「一键粘贴」**必须**从剪贴板字符串解码并导入：保存 API 地址、租户代码、API Key（复用保存路径，含 slug 校验）
- 字符串非 Base64 / 内容非法时**必须**返回明确错误、**禁止** panic

#### 场景：测试云端连接

**前提条件：**
- 用户在表单填写了 API 地址（API Key、租户代码可选）

**操作步骤：**
1. 用户点击"测试连接"按钮

**预期结果：**
- 系统**必须**测试**表单当前填写的值**（API 地址 / API Key / 租户代码），**无需**先点击"保存配置"
- 发送 GET 请求到 `{表单 api_url}/ping`
- 请求头携带 `Authorization: Bearer {key}`；其中 key 取表单 API Key，表单为空时**回落到已保存凭据**（支持「已存 Key、仅改其它项」时测试）
- **当** 表单填写了非空租户代码 **那么** 请求头**必须**额外携带 `X-Tenant-Id: {tenant}`；未填写时**禁止**携带该头
- 连接成功：显示"连接成功"，并展示服务端回显的已认证租户（`tenant`）；当服务端回显 `fallback=true` 时**必须**提示「凭据命中默认租户兜底（请确认 Key 归属）」——此为**信息提示、非租户不匹配**（连接成功时若申报非空则必等于解析租户，真正的不匹配由下方 403 路径捕获）
- 连接失败：显示具体错误信息：
  - 网络不可达："无法连接到服务器，请检查网络"
  - 401 错误："API Key 无效，请检查配置"
  - 403 且响应 `code` 为 `tenant_mismatch`：显示可识别的「租户代码与凭据归属不一致」文案，**禁止**吞为泛化错误
  - 其他错误：显示服务器返回的错误信息

#### 场景：执行全量同步

**前提条件：**
- 已配置云端 API
- 连接测试通过

**操作步骤：**
1. 用户点击"立即同步"按钮
2. 系统显示同步进度

**预期结果：**
- 发送 POST 请求到 `{api_url}/sync`
- 请求头携带 `Authorization: Bearer {api_key}`
- **当** 已配置非空租户代码 **那么** 请求头**必须**额外携带 `X-Tenant-Id: {tenant}`；未配置时**禁止**携带
- 请求体包含：
  - `client_id`: 客户端标识
  - `sync_time`: 同步时间戳
  - `base_version`: 本地持久化的云端基线版本（首次同步前为空/缺省时不携带，服务端按无条件覆盖处理）
  - `data`: 所有本地数据，与导出/导入范围一致，即**数据库全部业务表**：projects、cards、sf_senders、sf_orders、app_settings（若将来新增业务表，须同步加入导出、导入与云端同步的 data 中）
- 同步成功（200）：
  - 显示统计信息（项目数、卡片数等）
  - 记录上次同步时间
  - **必须**把本地持久化的 `base_version` 刷新为响应回传的 `server_version` 并落盘（否则下次同步会自发 409）
- 同步失败：显示具体错误信息

#### 场景：未配置租户代码时同步（兼容期）

**前提条件：**
- 已配置云端 API，但租户代码留空（含旧 `sync.toml` 升级后未填写的存量用户）

**操作步骤：**
1. 用户点击"立即同步"

**预期结果：**
- 客户端**禁止**发送 `X-Tenant-Id` 请求头
- 同步请求形态与未引入本能力前**逐字一致**，服务端缺头向后兼容放行、正常完成同步
- 系统**禁止**因租户代码为空而阻断同步（本能力为软约束，不做硬必填）

#### 场景：同步遇租户不匹配（403）

**前提条件：**
- 已配置非空租户代码，但申报的租户与写凭据解析出的租户不一致

**操作步骤：**
1. 用户点击"立即同步"

**预期结果：**
- 服务端返回 `403`、响应体 `code` 为 `tenant_mismatch`，本地数据与本地 `base_version` **均不被改动**
- 系统**必须**把该 403 解析为**可分辨的「租户不匹配」结果**（不并入笼统错误字符串、不与 401/409 混淆）
- 提示用户「申报的租户代码与 API Key 归属的租户不一致」，引导其修正租户代码或更换 API Key

#### 场景：同步认证失败

**前提条件：**
- API Key 无效或过期

**操作步骤：**
1. 用户点击"立即同步"

**预期结果：**
- 显示错误提示："认证失败，请检查 API Key"
- 本地数据不受影响

#### 场景：同步遇版本冲突（409）

**前提条件：**
- 本地持有的 `base_version` 落后于云端（其他设备已先同步）

**操作步骤：**
1. 用户点击"立即同步"

**预期结果：**
- 服务端返回 409，本地数据与本地 `base_version` **均不被改动**
- 系统**必须**把 409 解析为**可分辨的「版本冲突」结果**（不并入笼统错误字符串），并提供两条出路：
  - 「下载云端最新」：走从云端恢复（`/pull`）重建本地库后再决定
  - 「强制覆盖」：以 `force=true` 重新发起 `/sync`，用本机数据覆盖云端
- 冲突提示**应**展示云端当前 `server_version`（取自 409 响应体）
- 若 409 响应体解析失败（非 JSON / 缺 `server_version` 字段）：系统**仍必须**降级为可分辨的「版本冲突」结果（`server_version` 缺省为空），**禁止** panic 或回退为笼统报错，使用户仍能走「下载/强制覆盖」两条出路

#### 场景：清除同步配置

**操作步骤：**
1. 用户点击"清除配置"按钮
2. 系统显示确认对话框
3. 用户确认

**预期结果：**
- 清除 API 地址配置
- 清除 API Key
- 清除租户代码
- 清除上次同步时间
- 清除本地 `base_version`（下次同步将按首次处理）
- 保留 client_id（便于云端识别）
- 表单恢复空白状态
- 注（accepted-degraded，阶段 1 既有）：现状 `clear_sync_config` 删除整个 `sync.toml`（连带 client_id 一并丢失），与上「保留 client_id」措辞冲突；本期不扩范围改清除语义，仅确保「清除 `base_version`」「清除租户代码」不引入新矛盾（「清除租户代码」同样由删整文件实现，**不新增代码路径**），该不一致记为独立清理项（见 design 待解决问题）

### 需求：云端 API 规范

系统**必须**提供云端 API 规范文档，供用户自行实现接收服务。

#### 场景：API 规范文档内容

**规范包含：**

1. **认证方式**
   - API Key（Bearer Token）
   - 在请求头中携带：`Authorization: Bearer {api_key}`

2. **接口列表**
   - `GET /ping` - 连接测试
   - `POST /sync` - 全量同步

3. **连接测试接口**
   ```
   GET /ping
   Authorization: Bearer {api_key}

   响应（200）：
   {
     "success": true,
     "message": "pong",
     "server_time": "2026-01-23T14:30:00+08:00"
   }
   ```

4. **同步接口**
   ```
   POST /sync
   Authorization: Bearer {api_key}
   Content-Type: application/json

   请求体：
   {
     "client_id": "uuid",
     "sync_time": "2026-01-23T14:30:00+08:00",
     "data": {
       "projects": [...],
       "cards": [...],
       "sf_senders": [...],
       "sf_orders": [...]
     }
   }

   响应（200）：
   {
     "success": true,
     "message": "同步成功",
     "received_at": "2026-01-23T14:30:01+08:00",
     "stats": {
       "projects": 10,
       "cards": 500,
       "sf_senders": 5,
       "sf_orders": 100
     }
   }
   ```

5. **错误响应格式**
   ```json
   {
     "success": false,
     "message": "错误描述"
   }
   ```

6. **数据结构定义**
   - Project 结构
   - Card 结构
   - SFSender 结构
   - SFOrder 结构

7. **云端实现建议**
   - 按 `client_id` 隔离不同客户端数据
   - 实现请求频率限制
   - 使用 HTTPS
   - 验证 API Key 有效性

### 需求：桌面端从云端恢复同步数据

系统**必须**提供「从云端恢复」能力：以写入 Key 调用 `GET /pull` 拉回该租户全量快照，用快照**原子重建本地业务表**，并把本地 `base_version` 设为快照的 `server_version`。该操作会**销毁本地未上传的改动**，**必须**前置二次确认。当已配置非空租户代码时，`/pull` 请求**必须**携带 `X-Tenant-Id` 申报头供服务端交叉校验。

#### 场景：从云端恢复重建本地库

**前提条件：**
- 已配置云端 API 地址与写入 API Key

**操作步骤：**
1. 用户在"数据管理"页点击"从云端恢复"
2. 系统弹出确认对话框，明示「将用云端数据覆盖本地、丢失本地未上传改动」
3. 用户确认

**预期结果：**
- 发送 `GET /pull`，请求头携带 `Authorization: Bearer {api_key}`
- **当** 已配置非空租户代码 **那么** 请求头**必须**额外携带 `X-Tenant-Id: {tenant}`；未配置时**禁止**携带
- 用返回的 `data` 在**单个本地事务**内「**无条件清空** 5 张业务表 → 按快照写入」（projects、cards、sf_senders、sf_orders、app_settings），中途失败**必须**回滚、不留半重建状态
- **并且** 清空**必须无条件覆盖全部 5 张表（含 `app_settings`）**：即便快照 `app_settings` 为空数组也要清空本地旧配置，**禁止**沿用既有文件导入「app_settings 仅在 data 含该字段时才清」的条件清空（否则云端空配置时本地旧配置幽灵残留、`base_version` 已对齐但数据不一致）
- 恢复成功后把本地 `base_version` 设为返回的 `server_version` 并落盘
- 此后用户「立即同步」携带对齐的 `base_version`、应直接 200（不再 409）

#### 场景：恢复鉴权失败

**前提条件：**
- API Key 无效或缺失

**操作步骤：**
1. 用户点击"从云端恢复"

**预期结果：**
- `/pull` 返回 401
- 显示"认证失败，请检查 API Key"
- 本地数据**禁止**被改动（鉴权失败不进入重建流程）

#### 场景：恢复遇租户不匹配（403）

**前提条件：**
- 已配置非空租户代码，但申报的租户与写凭据解析出的租户不一致

**操作步骤：**
1. 用户点击"从云端恢复"

**预期结果：**
- `/pull` 返回 `403`、响应体 `code` 为 `tenant_mismatch`
- 系统**必须**显示可识别的「租户代码与凭据归属不一致」文案，**禁止**吞为泛化「拉取失败」
- 本地数据**禁止**被改动（不进入重建流程）

### 需求：桌面端同步模式与首启引导

系统**必须**支持两种客户端模式——**云同步**（已配置云端 API）与**纯本地**（仅本地 + 导出/导入备份）——并在首启引导用户选择、在标题栏常驻展示当前模式/租户身份。模式**必须**由 `api_url` 是否为空**派生**（非空=云同步、空=纯本地），**禁止**单独存储模式字段。**模式（展示）≠ 可同步就绪**：同步/恢复操作的可用性另由 `api_url` 非空**且**已配置 API Key（`has_api_key`）决定（`canSync`），其判据**禁止**简化为仅 `api_url`。首启网关**必须**是**阻断首屏**（加载 / 加载失败 / 网关 / 主界面互斥渲染，主内容视图在加载/网关期间**不挂载**），**禁止**为叠加在已挂载主界面上的遮罩。标题栏徽章**只展示**模式/租户、**禁止**作为归属判定依据（归属红线沿用既有「云端数据同步」需求，不另立第二份权威拷贝）。

#### 场景：首次启动显示模式选择网关

**前提条件：**
- `api_url` 为空（未配置云同步）
- onboarding「已选过模式」标志未设置

**操作步骤：**
1. 用户启动应用

**预期结果：**
- 系统**必须**显示应用内首屏遮罩网关（**非新窗口**），提供「云同步」与「纯本地」两个选择 + 简短模式说明
- 网关**禁止**可空手关闭（无关闭按钮、点遮罩外不关）——**必须**二选一才能进入应用
- 选「云同步」：**必须**导航到「租户 & 云端同步」设置页 + 置 onboarding 标志为已完成 + 关闭网关
- 选「纯本地」：**必须**置 onboarding 标志为已完成 + 关闭网关
- onboarding 标志记录的语义**必须**是「已做过一次模式选择」（点任一按钮即置位），**不是**「已配置云端」

#### 场景：onboarding 标志为设备本地

**前提条件：**
- 用户在某设备完成了模式选择

**预期结果：**
- onboarding 标志**必须**存于**设备本地存储**（不随云端同步），使每台设备独立 onboarding
- 标志**禁止**存于会同步到云端的设置（否则其它设备从未被引导却继承「已 onboard」，语义错）
- 标志**禁止**记录所选模式的具体值（模式由 `api_url` 派生，存模式为冗余真源）

#### 场景：存量用户升级不被错误打扰

**前提条件：**
- 用户从无本能力的旧版本升级（onboarding 标志不存在）

**预期结果：**
- **当** 存量用户已配置云同步（`api_url` 非空） **那么** 系统**必须**在启动时（配置加载完成后、网关判定之前）静默置 onboarding 标志为已完成，**不**显示网关（零打扰）
- **当** 存量用户为纯本地（`api_url` 空，**无论本地有无数据**） **那么** 允许显示一次网关；用户选「纯本地」后**必须**置标志、此后**不再**显示网关
- 触发判定**禁止**查询本地库数据量（无 count 调用）：全新用户与存量纯本地用户走**同一**路径，触发恒为 `!api_url && !onboardingDone`

#### 场景：标题栏常驻租户徽章

**前提条件：**
- 应用已进入主界面

**预期结果：**
- 标题栏**必须**常驻一个模式/租户徽章，按以下三态显示：
  - `api_url` 空 → 显示「纯本地」
  - `api_url` 非空且已申报租户 → 显示租户代码（如 `bh2ro`）
  - `api_url` 非空且未申报租户 → 显示「云同步」（**禁止**显示「未申报」等警示词——未申报是软约束兼容期的合法态）
- 点击徽章**必须**导航到「租户 & 云端同步」设置页
- 徽章**禁止**承担归属判断，仅作模式/身份展示

#### 场景：模式切换为配置变更的副产物

**前提条件：**
- 用户在主界面

**操作步骤：**
1. 纯本地用户在设置页填入 API 地址并保存 / 云同步用户清除云配置

**预期结果：**
- 系统**禁止**提供独立的「切换模式」控件
- 填入并保存非空 `api_url` 后，模式**必须**自动派生为云同步，徽章**必须**响应式更新为租户/「云同步」
- 清除云配置（`api_url` 变空）后，模式**必须**自动回到纯本地，徽章**必须**更新为「纯本地」
- 同步/恢复按钮的可用性**必须**用 `canSync`（`api_url` 非空 **且** `has_api_key`）判定，**禁止**仅用 `api_url`——仅填 api_url 未配 Key 时模式已是云同步、但同步/恢复仍不可用（后端硬要求 Key）

#### 场景：配置变更后徽章即时反映

**前提条件：**
- 徽章、网关、设置页、**卡片页**共享当前同步配置（响应式单一事实源）

**预期结果：**
- 共享状态**必须**持有完整同步配置字段（`api_url/tenant/has_api_key/last_sync_at/client_id/base_version`）——漏 `client_id`/`base_version` 会丢设置页展示/同步后更新的 UI 字段
- 保存/导入云配置成功后，系统**必须**用**命令返回的配置响应**（含后端规整后的 tenant）就地更新共享状态（无需重新拉取）；徽章**必须**立即反映
- **同步/恢复成功后**，系统**必须**把新 `last_sync_at`/`base_version` 回写共享状态（否则展示漂移）
- 清除配置后**必须**重置共享状态、但**保留 `client_id`**（设备标识不随清云配丢失）
- 徽章判据**必须**取**已保存**配置（共享状态），**禁止**取表单草稿——用户在表单编辑 api_url 未点保存时徽章/模式**不变**
- 共享状态**必须**在应用启动加载**完成后**再渲染徽章与判定网关（防首帧默认值闪烁）；配置加载**失败**（命令 reject）时**禁止**按「空 api_url」判首启或静默置 onboarding 标志，**必须**显示错误 + 重试，加载状态最终落定不卡死首屏

#### 场景：选「云同步」后中途放弃配置

**前提条件：**
- 全新用户在网关点「云同步」、被导航到设置页，但未填写 `api_url` 就离开

**预期结果：**
- onboarding 标志**已在点「云同步」时置位**（标志语义=已选过模式、非已配置）
- 下次启动：`!api_url && onboardingDone` → **不再**显示网关，以纯本地态静默进入，徽章显「纯本地」
- 本次会话内徽章即显「纯本地」是**正确**的（模式派生自已保存 `api_url`、未配置即本地态），非缺陷；用户点徽章可重进设置页续配

#### 场景：网关期间主界面零副作用、首启副作用不被吞

**前提条件：**
- 首启显示网关（阻断首屏）

**预期结果：**
- 网关期间主内容视图（含卡片页）**禁止**挂载——避免空项目时卡片页弹「新建项目」对话框等副作用盖在网关下
- 既有「无打印机→跳打印机配置页」逻辑**必须**抽为可复用函数，仅在**无网关分支**或**网关选「纯本地」关闭后**执行；网关选「云同步」分支**禁止**触发它（否则覆盖已导航的设置页）
- 更新检查（静默检查 + 定时器）**必须**与网关正交、无条件运行，**禁止**被「网关成立则提前返回」吞掉；**主界面未挂载的任一态期间（加载占位 / 网关 / 加载失败错误分支）禁止**弹出可绕过它（如跳转「关于」页，目标视图未挂载会是死按钮）的更新通知

#### 场景：卡片页同步状态收口共享源

**前提条件：**
- 卡片页（CardManagementView）依据同步配置 gate 其同步/恢复按钮

**预期结果：**
- 卡片页的同步配置状态**必须**取自共享单一事实源（同徽章/设置页），**禁止**自行独立加载一份
- 在设置页 `clear` 云配置后，卡片页同步按钮**必须**随共享状态即时失效（不得停留在旧「已配置」态直到重进页面）
