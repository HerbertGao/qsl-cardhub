## 修改需求

### 需求：云端数据同步

系统**必须**支持将本地数据全量同步到用户自建的云端 API，并参与服务端的乐观并发版本护栏（见 `cloud-sync-versioning`）：持久化并回传 `base_version`、成功后刷新基线、遇 409 引导用户下载或强制覆盖。

系统**必须**支持用户申报所属租户（tenant 代码），并在云端请求中以请求头 `X-Tenant-Id` 携带该申报值，供服务端交叉校验（见 `tenant-isolation` / `cloud-backend-api`）。租户归属的真源**永远**是写凭据（`key→tenant`），`X-Tenant-Id` 仅用于申报与校验，**禁止**被客户端或服务端当作写入/读取的归属目标。租户代码为**软约束**：未填写时客户端**禁止**发送 `X-Tenant-Id`，行为与未引入本能力前逐字一致（服务端缺头向后兼容放行）。

#### 场景：配置云端 API

**前提条件：**
- 用户已部署云端接收服务（或选用官方云）
- 用户已获取 API 地址和 API Key

**操作步骤：**
1. 用户打开"数据管理"页面
2. 用户在"云端同步"区域的「API 地址」下拉框选择**官方云预设**或**手动输入**自定义 API 地址
3. 用户输入 API Key
4. 用户输入租户代码（可选；输入完毕失焦时自动格式化）
5. 用户点击"保存配置"

**预期结果：**
- 「API 地址」**必须**是可手动输入的下拉框，预设至少两项：空值（未配置）与官方云地址；选择预设或手填均可
- 选中**官方云预设**时，租户代码与 API Key **必须**为必填（API Key 已保存时可留空保持不变）；其它地址（自托管）保持软约束、不强制
- API 地址保存到配置文件
- API Key 加密存储（使用凭据加密存储系统）；存储键名：`qsl-cardhub:sync:api_key`
- 租户代码保存到配置文件（`sync.toml` 的 `tenant` 字段）；留空时该字段为空/缺省，旧 `sync.toml`（无 `tenant` 字段）解析后**必须**为空（向后兼容）
- 显示保存成功提示
- 首次保存时自动生成 client_id（UUID 格式）

#### 场景：租户代码输入完毕自动格式化

**前提条件：**
- 用户在配置表单的租户代码输入框输入内容

**操作步骤：**
1. 用户输入含大写字母/空格/非法字符/超长的内容（如 `BH2RO`、`My Tenant`）
2. 输入框失焦（输入完毕）

**预期结果：**
- 系统**必须**在失焦时自动把租户代码格式化为合法 slug：去首尾空白 → 转小写 → 去除非 `[a-z0-9-]` 字符 → 截断至 32 位（如 `BH2RO`→`bh2ro`、`My Tenant`→`mytenant`）
- 格式化为**可见**变更（当面发生于输入框），不是保存时的静默转换
- 命令层保存时**仍必须**以 `^[a-z0-9-]{1,32}$` 做权威校验作兜底（格式化后应恒通过）

#### 场景：导出/导入同步配置（设备间迁移）

**前提条件：**
- 用户已配置云端 API（含 API Key）

**操作步骤：**
1. 用户点击"一键复制"
2. 在另一处/设备点击"一键粘贴"

**预期结果：**
- 「一键复制」**必须**把当前 API 地址、租户代码、API Key 导出为 Base64 编码字符串并写入剪贴板；导出串**含明文 API Key**（Base64 仅编码、非加密），界面**必须**提示其敏感性、仅供本机设备间迁移
- 导出**禁止**包含 `client_id`（设备本地自动生成、不随配置迁移）
- 「一键粘贴」**必须**从剪贴板字符串解码并导入：保存 API 地址、租户代码、API Key（复用保存路径，含 slug 校验）
- 字符串非 Base64 / 内容非法时**必须**返回明确错误、**禁止** panic

#### 场景：测试云端连接

**前提条件：**
- 用户在表单填写了 API 地址（API Key、租户代码可选）

**操作步骤：**
1. 用户点击"测试连接"按钮

**预期结果：**
- 系统**必须**测试**表单当前填写的值**（API 地址 / API Key / 租户代码），**无需**先点击"保存配置"
- 发送 GET 请求到 `{表单 api_url}/ping`
- 请求头携带 `Authorization: Bearer {key}`；其中 key 取表单 API Key，表单为空时**回落到已保存凭据**（支持「已存 Key、仅改其它项」时测试）
- **当** 表单填写了非空租户代码 **那么** 请求头**必须**额外携带 `X-Tenant-Id: {tenant}`；未填写时**禁止**携带该头
- 连接成功：显示"连接成功"，并展示服务端回显的已认证租户（`tenant`）；当服务端回显 `fallback=true` 时**必须**提示「凭据命中默认租户兜底（请确认 Key 归属）」——此为**信息提示、非租户不匹配**（连接成功时若申报非空则必等于解析租户，真正的不匹配由下方 403 路径捕获）
- 连接失败：显示具体错误信息：
  - 网络不可达："无法连接到服务器，请检查网络"
  - 401 错误："API Key 无效，请检查配置"
  - 403 且响应 `code` 为 `tenant_mismatch`：显示可识别的「租户代码与凭据归属不一致」文案，**禁止**吞为泛化错误
  - 其他错误：显示服务器返回的错误信息

#### 场景：执行全量同步

**前提条件：**
- 已配置云端 API
- 连接测试通过

**操作步骤：**
1. 用户点击"立即同步"按钮
2. 系统显示同步进度

**预期结果：**
- 发送 POST 请求到 `{api_url}/sync`
- 请求头携带 `Authorization: Bearer {api_key}`
- **当** 已配置非空租户代码 **那么** 请求头**必须**额外携带 `X-Tenant-Id: {tenant}`；未配置时**禁止**携带
- 请求体包含：
  - `client_id`: 客户端标识
  - `sync_time`: 同步时间戳
  - `base_version`: 本地持久化的云端基线版本（首次同步前为空/缺省时不携带，服务端按无条件覆盖处理）
  - `data`: 所有本地数据，与导出/导入范围一致，即**数据库全部业务表**：projects、cards、sf_senders、sf_orders、app_settings（若将来新增业务表，须同步加入导出、导入与云端同步的 data 中）
- 同步成功（200）：
  - 显示统计信息（项目数、卡片数等）
  - 记录上次同步时间
  - **必须**把本地持久化的 `base_version` 刷新为响应回传的 `server_version` 并落盘（否则下次同步会自发 409）
- 同步失败：显示具体错误信息

#### 场景：未配置租户代码时同步（兼容期）

**前提条件：**
- 已配置云端 API，但租户代码留空（含旧 `sync.toml` 升级后未填写的存量用户）

**操作步骤：**
1. 用户点击"立即同步"

**预期结果：**
- 客户端**禁止**发送 `X-Tenant-Id` 请求头
- 同步请求形态与未引入本能力前**逐字一致**，服务端缺头向后兼容放行、正常完成同步
- 系统**禁止**因租户代码为空而阻断同步（本能力为软约束，不做硬必填）

#### 场景：同步遇租户不匹配（403）

**前提条件：**
- 已配置非空租户代码，但申报的租户与写凭据解析出的租户不一致

**操作步骤：**
1. 用户点击"立即同步"

**预期结果：**
- 服务端返回 `403`、响应体 `code` 为 `tenant_mismatch`，本地数据与本地 `base_version` **均不被改动**
- 系统**必须**把该 403 解析为**可分辨的「租户不匹配」结果**（不并入笼统错误字符串、不与 401/409 混淆）
- 提示用户「申报的租户代码与 API Key 归属的租户不一致」，引导其修正租户代码或更换 API Key

#### 场景：同步认证失败

**前提条件：**
- API Key 无效或过期

**操作步骤：**
1. 用户点击"立即同步"

**预期结果：**
- 显示错误提示："认证失败，请检查 API Key"
- 本地数据不受影响

#### 场景：同步遇版本冲突（409）

**前提条件：**
- 本地持有的 `base_version` 落后于云端（其他设备已先同步）

**操作步骤：**
1. 用户点击"立即同步"

**预期结果：**
- 服务端返回 409，本地数据与本地 `base_version` **均不被改动**
- 系统**必须**把 409 解析为**可分辨的「版本冲突」结果**（不并入笼统错误字符串），并提供两条出路：
  - 「下载云端最新」：走从云端恢复（`/pull`）重建本地库后再决定
  - 「强制覆盖」：以 `force=true` 重新发起 `/sync`，用本机数据覆盖云端
- 冲突提示**应**展示云端当前 `server_version`（取自 409 响应体）
- 若 409 响应体解析失败（非 JSON / 缺 `server_version` 字段）：系统**仍必须**降级为可分辨的「版本冲突」结果（`server_version` 缺省为空），**禁止** panic 或回退为笼统报错，使用户仍能走「下载/强制覆盖」两条出路

#### 场景：清除同步配置

**操作步骤：**
1. 用户点击"清除配置"按钮
2. 系统显示确认对话框
3. 用户确认

**预期结果：**
- 清除 API 地址配置
- 清除 API Key
- 清除租户代码
- 清除上次同步时间
- 清除本地 `base_version`（下次同步将按首次处理）
- 保留 client_id（便于云端识别）
- 表单恢复空白状态
- 注（accepted-degraded，阶段 1 既有）：现状 `clear_sync_config` 删除整个 `sync.toml`（连带 client_id 一并丢失），与上「保留 client_id」措辞冲突；本期不扩范围改清除语义，仅确保「清除 `base_version`」「清除租户代码」不引入新矛盾（「清除租户代码」同样由删整文件实现，**不新增代码路径**），该不一致记为独立清理项（见 design 待解决问题）

### 需求：桌面端从云端恢复同步数据

系统**必须**提供「从云端恢复」能力：以写入 Key 调用 `GET /pull` 拉回该租户全量快照，用快照**原子重建本地业务表**，并把本地 `base_version` 设为快照的 `server_version`。该操作会**销毁本地未上传的改动**，**必须**前置二次确认。当已配置非空租户代码时，`/pull` 请求**必须**携带 `X-Tenant-Id` 申报头供服务端交叉校验。

#### 场景：从云端恢复重建本地库

**前提条件：**
- 已配置云端 API 地址与写入 API Key

**操作步骤：**
1. 用户在"数据管理"页点击"从云端恢复"
2. 系统弹出确认对话框，明示「将用云端数据覆盖本地、丢失本地未上传改动」
3. 用户确认

**预期结果：**
- 发送 `GET /pull`，请求头携带 `Authorization: Bearer {api_key}`
- **当** 已配置非空租户代码 **那么** 请求头**必须**额外携带 `X-Tenant-Id: {tenant}`；未配置时**禁止**携带
- 用返回的 `data` 在**单个本地事务**内「**无条件清空** 5 张业务表 → 按快照写入」（projects、cards、sf_senders、sf_orders、app_settings），中途失败**必须**回滚、不留半重建状态
- **并且** 清空**必须无条件覆盖全部 5 张表（含 `app_settings`）**：即便快照 `app_settings` 为空数组也要清空本地旧配置，**禁止**沿用既有文件导入「app_settings 仅在 data 含该字段时才清」的条件清空（否则云端空配置时本地旧配置幽灵残留、`base_version` 已对齐但数据不一致）
- 恢复成功后把本地 `base_version` 设为返回的 `server_version` 并落盘
- 此后用户「立即同步」携带对齐的 `base_version`、应直接 200（不再 409）

#### 场景：恢复鉴权失败

**前提条件：**
- API Key 无效或缺失

**操作步骤：**
1. 用户点击"从云端恢复"

**预期结果：**
- `/pull` 返回 401
- 显示"认证失败，请检查 API Key"
- 本地数据**禁止**被改动（鉴权失败不进入重建流程）

#### 场景：恢复遇租户不匹配（403）

**前提条件：**
- 已配置非空租户代码，但申报的租户与写凭据解析出的租户不一致

**操作步骤：**
1. 用户点击"从云端恢复"

**预期结果：**
- `/pull` 返回 `403`、响应体 `code` 为 `tenant_mismatch`
- 系统**必须**显示可识别的「租户代码与凭据归属不一致」文案，**禁止**吞为泛化「拉取失败」
- 本地数据**禁止**被改动（不进入重建流程）
