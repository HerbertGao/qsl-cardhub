## 修改需求

### 需求：用户体验

查询侧防爬**必须**对正常用户尽量无感：PoW 计算**必须**在后台（Web Worker）进行、不阻塞 UI，且自适应难度下正常用户耗时 ~0.1–0.3s（手机端无感）；会话建立与查询签名**必须**对用户透明（无需人工交互）。**不再有算术验证码交互**（订阅与查询均不弹验证码）。

#### 场景：用户执行查询与订阅操作

- **当** 用户执行查询或订阅操作
- **那么** 全程**禁止**出现需人工作答的验证码交互
- **并且** PoW **必须**后台计算、不阻塞 UI，正常用户基本无感（手机 ~0.1–0.3s）
- **并且** 会话建立、查询签名对用户透明

### 需求：性能

限流检查延迟**必须** < 10ms（KV 读取）；查询会话签名校验**必须** < 5ms（纯计算 + 一次 KV 读会话）；PoW 验证（服务端，单次 sha256 + KV 校删）**必须**为常数级低延迟。客户端 PoW 计算耗时由自适应难度约束在正常用户 ~0.1–0.3s。

#### 场景：处理防护相关请求

- **当** 系统执行限流检查、会话签名校验或 PoW 验证
- **那么** 限流检查 < 10ms、会话签名校验 < 5ms、PoW 服务端验证为常数级低延迟
- **并且** 正常用户客户端 PoW 耗时受自适应难度约束在 ~0.1–0.3s

### 需求：安全

会话签名密钥（`sk`）**必须**为**每会话专属、短时**，经 `POST /api/session` 响应体下发，**禁止**经 `/api/config` 或任何静态途径下发可公开的查询签名密钥（静态 `CLIENT_SIGN_KEY` 退役、不再用于查询签名）。会话 HMAC 密钥与 PoW/会话相关服务端密钥**必须**仅存于 Secret（如 `SESSION_SECRET`）。系统**禁止**记录用户 IP 的完整日志，IP 仅用于限流/防爬计数。

#### 场景：处理密钥与用户 IP

- **当** 系统签发会话签名密钥、处理服务端密钥或用户 IP
- **那么** 会话签名密钥**必须**每会话专属短时、仅经会话响应下发；**禁止**静态下发查询签名密钥
- **并且** 服务端密钥仅存于 Secret；**禁止**记录用户 IP 完整日志（仅用于计数）

### 需求：范围限制

本能力禁止包含以下内容：图形验证码（点选、滑动拼图）、第三方验证码服务集成（腾讯天御、极验等）、行为分析 / 风控系统、IP 黑名单持久化管理。

查询侧 PoW（无状态计算成本，非行为画像）、短时会话、会话配额、按真实 IP 自适应难度**属本能力范围内**（语义见 `query-antibot-session`），**不**视为被禁的「行为分析 / 风控系统」；上述其余禁止项不变。

#### 场景：评估超出范围的防护手段

- **当** 评估是否引入图形验证码、第三方验证码服务、行为分析/风控系统或 IP 黑名单持久化管理
- **那么** 这些手段均不在本能力范围内，不予实现
- **并且** 查询侧 PoW、短时会话、会话配额、按真实 IP 自适应难度属本能力范围内，**不**视为被禁的行为分析/风控系统

## 移除需求

### 需求：算术验证码

**Reason**: 算术验证码为「剧场」——仅在订阅入口弹出、不防护查询本身且可绕过；由 PoW + 短时会话体系统一替换（见 `query-antibot-session`）。
**Migration**: **查询侧**防自动化由 PoW + 会话 + 配额承担（见 `query-antibot-session`）；**订阅入口**仅移除算术验证码交互（点击直接进入微信 OAuth 授权），其 `/api/wechat/auth-callback` 回调**仍只按既有 IP 限流 + 微信 OAuth code 约束**处理、**不**升级为 PoW+会话（见非目标）；前端移除验证码交互。

### 需求：验证码生成接口

**Reason**: `GET /api/captcha` 随算术验证码一并移除。
**Migration**: 改用 `GET /api/session/challenge`（PoW 题）+ `POST /api/session`（验 PoW 签发会话），见 `query-antibot-session`。

### 需求：验证码校验参数

**Reason**: 订阅不再携带 `captcha_token`/`captcha_answer`（算术验证码移除）。
**Migration**: 自动化防护改由会话机制承担；订阅入口不再需要验证码参数。

### 需求：请求签名校验

**Reason**: 基于静态可公开 `CLIENT_SIGN_KEY` 的签名对防爬零收益（攻击者直接持公开 key 签名）；由会话专属动态签名取代。
**Migration**: 查询签名改用会话专属 `sk`（见 `query-antibot-session`「查询侧会话校验与动态签名」），`_ts`/`_nonce`/时窗/nonce 防重放机制保留但密钥改为 `sk`。

### 需求：请求签名格式

**Reason**: 签名密钥从静态 `CLIENT_SIGN_KEY` 改为会话专属 `sk`，旧的 `payload + 密钥` 取 SHA-256 摘要的格式定义失效。
**Migration**: 新签名格式见 `query-antibot-session`：`_sig = HMAC-SHA256(sk, canonicalPayload)`（`canonicalPayload` 定义见 `query-antibot-session`「查询侧会话校验与动态签名」；**必须** HMAC，**禁止**对 `payload + sk` 取摘要的旧式），并额外携带会话 token。
