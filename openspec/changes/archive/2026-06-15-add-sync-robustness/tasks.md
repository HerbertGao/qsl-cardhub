## 1. Worker：/sync 乐观并发护栏（OCC）

- [x] 1.1 解析请求体新增可选字段：`base_version`（整数，缺省/非整数视为「未携带」）与 `force`（布尔，仅 `true` 生效）；保留现有 `client_id`/`data` 形态校验（缺失 400）不变
- [x] 1.2 实现**守卫路径**（携带 `base_version` 且非 `force`）：在单个 `DB.batch` 内（**占位符一律用位置匿名 `?` + `.bind()` 顺序绑定，禁混用 `?1/?2` 与 `?`**——SQLite 中二者共享编号命名空间会错位）——5 张业务表 `DELETE … WHERE tenant_id=? AND (SELECT server_version FROM sync_meta WHERE tenant_id=?)=?`（`.bind(tenant_id, tenant_id, base_version)`）；全部 `INSERT` 改 `INSERT INTO … SELECT ?,?,… WHERE (SELECT server_version FROM sync_meta WHERE tenant_id=?)=?`（`.bind(<行各列值…>, tenant_id, base_version)`）；**最后一条**为 CAS `UPDATE sync_meta SET server_version=server_version+1, last_client_id=?, sync_time=?, received_at=? WHERE tenant_id=? AND server_version=?`（`.bind(last_client_id, sync_time, received_at, tenant_id, base_version)`）。**必须移除**现有 `INSERT INTO sync_meta … ON CONFLICT … DO UPDATE`（index.js 约 406-412 行）——由本路径末条 CAS 取代；**保证整个 batch 内只有一条 sync_meta 写语句、且为 stmts 数组的最后一个元素**（实现需保证 push 顺序：DELETE×5 → INSERT×N → 最后 push CAS），否则 1.3 的「读末条结果」会读错语句
- [x] 1.3 **守卫路径** 409 判定（**仅守卫路径走此判定；无条件路径见 1.4，永不 409**）：守卫路径 batch 的**最后一条语句即 CAS**（push 顺序 DELETE×5→INSERT×N→CAS），读**该 CAS 对应结果**的 `result.meta.changes`（D1 batch 返回数组按语句位置一一对应，取 CAS 那条；**禁** `SELECT changes()`：`result.meta.changes` 按语句精确归属，`SELECT changes()` 读「最近 DML」在多语句 batch 中归属易错位）——`===1` 走 200、`===0` 走 409；409 时**禁止**已对数据产生改动（由 1.2 守卫 + 末条 CAS 的结构保证）；**409 体的云端当前 `server_version` 来源**：CAS 结果只给 `changes`、不给当前值，故 409 分支**必须**在 batch 提交后补一次 `SELECT server_version FROM sync_meta WHERE tenant_id=?`（`.first()`）读取回传（仅 409 路径多此 1 次读；batch 已零改动提交、此后读安全）；**若 sync_meta 行不存在**（守卫读得 NULL → CAS 也 0 行 → 409，补读 `.first()` 返回 `null`），取值**必须**用 `row?.server_version ?? null`（**禁** `row.server_version`——`row` 为 `null` 时抛错→落顶层 catch→500、而非契约要求的 409），409 体 `server_version` **必须**回 `null`（或省略该字段），**禁止**回 `undefined`/`NaN`（否则客户端 `base_version` 被写坏）
- [x] 1.4 实现**无条件路径**（`force=true` 或未携带 `base_version`）：`DELETE×5`（仅 `WHERE tenant_id=?`）+ `INSERT×N`（无守卫）+ upsert `INSERT INTO sync_meta(tenant_id,server_version,last_client_id,sync_time,received_at) VALUES(?,1,?,?,?) ON CONFLICT(tenant_id) DO UPDATE SET server_version=sync_meta.server_version+1, last_client_id=excluded.last_client_id, sync_time=excluded.sync_time, received_at=excluded.received_at`，全在单 `DB.batch`；**该 upsert 是本路径唯一的 sync_meta 写、取代现有 406-412 的 upsert**；本路径 upsert 永远成功（changes≥1）、**不判 409**；版本读回见 1.5（upsert **之后**再追加一条 SELECT，故 upsert **不是**数组末元素）
- [x] 1.5 200 响应体加 `server_version`（新版本）：**守卫路径**直接用 `base_version+1`（确定，无需读回）；**无条件路径在同一 `DB.batch` 的 upsert 之后追加一条 `SELECT server_version FROM sync_meta WHERE tenant_id=?`**（成为该路径数组末元素），读其结果取新版本（同事务 read-your-writes，消除 batch 外再读与并发写的微竞态）。两路径「读哪条结果」不同：守卫读 CAS 那条（1.3）、无条件读末条 SELECT；**实现必须在 push 各语句时记录目标结果索引（如 `casIndex`/`readbackIndex`）、按索引取 `results[idx]`，禁用 `results[results.length-1]` 这类「读末元素」共用 helper**（守卫路径 INSERT×N 改变数组长度、共用 helper 会随之静默错位）
- [x] 1.6 复核单 batch **语句条数**在 D1 上限内：**D1 batch 内每条 prepared statement 各计 1 次查询，上限 1000(Paid)/50(Free) 是单次 Worker invocation 的查询总数（与单条语句行数无关）**；语句条数（batch 内）：**守卫 /sync = `6+N`**（5 DELETE+N INSERT+1 CAS；409 分支再 +1 次 batch 后 SELECT）、**无条件 /sync = `7+N`**（多 1 条读回 SELECT）、**/pull = 6**（5 业务表+1 sync_meta SELECT）；**单次 Worker invocation 查询总数还须 + `resolveTenant`**（表驱动 +1、兜底再 +1 计数器 UPDATE）≈ batch 语句数 +1~2，N=业务总行数（cards 415 → ~423/424 条，Paid 1000 内有余量，数据翻倍即逼近，须留意）；单行守卫 INSERT 绑定参数 ≈ 列数+2（cards 13 个）远低于「每查询 100 绑定参数」上限；保留「超限须分块且跨块非原子」为 out-of-scope（阶段后续），本期单 batch 处理
- [x] 1.7 错误/409 响应**禁止**回显内部结构（沿用阶段 0 脱敏）；`base_version` 校验**必须**严格 `Number.isInteger(base_version)`，否则按「未携带」降级走无条件路径（**禁** `parseInt`：防 `"5abc"` 部分解析、`5.7`/`true`/数字字符串歧义）；**已知边界**：数字字符串（如 `"5"`）也判非整数→静默降级为无条件覆盖、护栏对该请求不生效——本项目桌面端 Rust 发整数 `base_version`（`Option<i64>` 序列化为数字）不受影响，第三方/代理客户端**必须**发整数；本期不启用 D1 read replication / Sessions API，全部读写走 primary（守卫 read-your-writes 与 /pull 一致性的前提，见 design）
