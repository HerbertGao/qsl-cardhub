## ADDED Requirements

### 需求：GitHub Release 资产可用性验证

在通过 GitHub Releases API 检测到新版本后，必须验证该 Release 中是否已包含当前平台所需的安装包资产文件，才能判定更新可用。

#### 场景：Release 中包含当前平台的安装包

- **当** GitHub Releases API 返回的最新 Release 版本号高于当前版本
- **并且** Release 的 `assets` 列表中包含当前平台对应的安装包文件
- **那么** 必须判定为"有更新可用"
- **并且** 必须正常展示更新信息

#### 场景：Release 中尚未包含当前平台的安装包

- **当** GitHub Releases API 返回的最新 Release 版本号高于当前版本
- **并且** Release 的 `assets` 列表中不包含当前平台对应的安装包文件
- **那么** 必须判定为"暂无更新"（而非"有更新可用"）
- **并且** 静默检查时不显示任何提示
- **并且** 手动检查时必须显示"已是最新版本"（与真正无更新时行为一致）

#### 场景：Release 中无任何资产文件

- **当** GitHub Releases API 返回的最新 Release 版本号高于当前版本
- **并且** Release 的 `assets` 列表为空
- **那么** 必须判定为"暂无更新"
- **并且** 行为与"已是最新版本"一致

---

### 需求：平台安装包文件名匹配规则

系统必须根据当前操作系统和架构，匹配 Release 资产中对应的安装包文件名。

#### 场景：macOS 平台的安装包匹配

- **当** 当前平台为 macOS
- **那么** 必须在 Release 资产中查找文件名包含 `.dmg` 的文件
- **并且** macOS ARM64 必须匹配包含 `macos-arm64` 的文件名
- **并且** macOS x64 必须匹配包含 `macos-x64` 的文件名

#### 场景：Windows 平台的安装包匹配

- **当** 当前平台为 Windows
- **那么** 必须在 Release 资产中查找文件名包含 `-setup.exe` 的文件（NSIS 安装包）
- **并且** Windows x64 必须匹配包含 `windows-x64` 的文件名
- **并且** Windows ARM64 必须匹配包含 `windows-arm64` 的文件名

#### 场景：无法确定当前平台

- **当** 无法识别当前操作系统或架构
- **那么** 必须跳过资产可用性验证
- **并且** 必须回退到仅基于版本号的判断逻辑（保持向后兼容）

---

### 需求：Tauri Updater 更新不可用处理

当 Tauri Updater 报告有更新但实际下载端点尚不可用时，必须优雅处理而非报错。

#### 场景：Tauri Updater 报告有更新但下载失败

- **当** Tauri Updater 的 `check()` 返回有更新可用
- **并且** 尝试获取更新详情或下载时发生网络错误（如 404、连接超时）
- **那么** 必须将此情况视为"暂无更新"
- **并且** 必须在控制台记录日志（warn 级别）说明原因
- **并且** 静默检查时不显示任何错误提示
- **并且** 手动检查时必须显示"已是最新版本"

#### 场景：Tauri Updater 正常返回有更新

- **当** Tauri Updater 的 `check()` 返回有更新可用
- **并且** 更新信息（版本号、下载地址等）均有效
- **那么** 必须正常展示更新信息（行为不变）
